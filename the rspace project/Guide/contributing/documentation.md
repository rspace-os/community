# Documentation Contributions

Good documentation makes RSpace accessible to everyone. There are several areas where contributions are welcome, each with a slightly different home and process.

## Types of Documentation

### User Documentation
Guides, tutorials, and reference material for researchers using RSpace — covering features, workflows, integrations, and best practices.

→ **[documentation.researchspace.com](https://documentation.researchspace.com)**

🚧 We're still working out a community contribution workflow for user docs. In the meantime, if you'd like to contribute — whether fixing an error, adding a missing guide, or documenting something you know well — please **[raise an issue on rspace-web](https://github.com/rspace-os/rspace-web/issues)** or reach out at opensource@researchspace.com and we'll work with you from there.

### Developer Documentation
Technical documentation for contributors: setup guides, architecture notes, coding standards, and development workflows.

→ **[DevDocs on GitHub](https://github.com/rspace-os/rspace-web/tree/main/DevDocs)**

Developer docs live directly in the rspace-web repository. If you're contributing code and want to improve or extend the developer docs alongside it, that's very welcome — just include it in your pull request.

### API Documentation
Reference documentation for the RSpace REST API, including endpoint specs, authentication, and usage examples.

→ **[API Reference](https://community.researchspace.com/public/apiDocs)**

If you spot gaps, errors, or missing examples in the API docs, please **[raise an issue on rspace-web](https://github.com/rspace-os/rspace-web/issues)**. For improvements that come alongside API code changes, include the documentation update in your pull request.

### This Playbook
The guide you're reading is itself open to contributions. If something is unclear, missing, or out of date, you're welcome to raise an issue or open a pull request directly on the community repository.
